## 안건 및 논의 내용

### 안건 1: 에디터의 최종 비전 — "뷰어 + 확장팩"

#### 현황

- 에디터는 React 컴포넌트 기반으로 동적 웹 페이지를 시각적으로 구성 가능
- JSX Export/Import 기능 구현 완료, 폼 렌더링 테스트까지 진행
- 서버 로직 삽입(API 호출 등)은 아직 미구현

#### 논의 내용

- **Attendees 1**: 두 가지 컨셉을 합쳐야 한다. (1) 독립된 뷰어 컴포넌트, (2) 내부 서비스 로직을 넣을 수 있는 "확장팩". iframe으로 들어가면 내부 앱처럼 작동 가능. 예시: 크레딧 사용량 조회 → 현재는 서버에서 보여주고 사람이 후속 조치 → 에디터 결과물이 클라이언트 툴이 되면 파라미터로 직접 보여주고 필요한 정보만 입력하면 됨.
- **Attendees 5 (다봄)**: 현재까지 폼 관련 로직만 테스트. 서버 호출 같은 동적 로직은 아직 안 됨.
- **Attendees 1**: 로직 삽입에는 다이나믹 번들링 필요 (Sandpack 또는 유사 도구). 허용된 API만 사용하도록 강하게 제한해야 함. 워크플로우까지 가면 복잡도 급증.

#### 결정 사항

- "뷰어 컴포넌트 + 확장팩" 구조를 기본 방향으로 채택
- 다봄의 기존 에디터 프레임워크를 베이스로 유지, 크게 벗어나지 않는 선에서 확장
- 로직 삽입은 탐색 영역 — 허용된 API/기능만 사용 가능하도록 설계

### 안건 2: 입력 채널 — Figma / Stitch / Chat

#### 논의 내용

- **Attendees 1**: 입력 채널은 3개 — Figma, Stitch(확인 필요), Chat. 편집을 고려하면 다봄 방식(React DOM 에디터)이 유리하므로 이 스펙대로 뷰를 만드는 것이 첫 단계.
- **Attendees 3**: 뷰만? 피그마든 뭐든 레이어별로 컨버팅되어 화면에 그려지는 것?
- **Attendees 1**: 맞다. 최초는 통째로 갖고 오고, 이후는 에디터에서 추가 편집. 피그마→에디터 방향만 가능 (역방향 export는 안 됨).
- **Attendees 2**: CMS와 커플링되는 순간 피그마에서 들어오기 어려운 구조가 될 것. 한번 에디터로 오면 여기서 작업하는 게 맞음.
- **Attendees 5 (다봄)**: 여기서 수정하고 핸들링하는 게 낫지 않을까?
- **Attendees 2**: Pencel.dev 같은 Design-to-Code 도구 공유 예정, 참고할 것

#### 결정 사항

- 최초 디자인은 Figma 등에서 가져오되, 이후 편집은 에디터에서 진행
- Figma→에디터 단방향 흐름 (역방향 미지원)
- 일부 요소만 가져오는 방법도 향후 검토 필요

### 안건 3: 삼성닷컴 로컬라이제이션 PoC

#### 현황

- 삼성닷컴 관련 구두 논의 진행 중 (계약 전)
- 1차 PoC: 삼성 시스템에는 붙이지 않고, 기술적으로 "되는가" 검증

#### 논의 내용

- **Attendees 3**: 삼성이 원하는 목표는? 로컬라이제이션(언어 변환) + 특장점(현지 특성) 변환이면 끝?
- **Attendees 1**: 언어 + 특징. 가이드라인이 정해져 있고 그에 따라 변환. 에디터의 섹션 구조를 활용하면 섹션별 로컬라이제이션 설정 가능. 프롬프트로 섹션 업데이트도 가능할 것.
- **Attendees 2**: PoC이므로 "되는가"가 핵심. 로컬라이제이션은 대부분 될 것으로 추론. 오히려 Design-to-Code가 어려운 영역. 스텝 1: Design-to-Code, 스텝 2: 로컬라이제이션.
- **Attendees 2**: 삼성은 계약 과정이 오래 걸림. 하반기까지 진행될 것.
- **Attendees 5 (다봄)**: 디자인 토큰으로 정리하면 되지 않을까? (테마 관련)

#### 결정 사항

- 삼성 PoC는 급하지 않음 (하반기 이후)
- 스텝 1: Design-to-Code 파이프라인, 스텝 2: 로컬라이제이션
- 고객사별 테마(폰트, 색상, 로고 등) 사전 정의 필요

### 안건 4: 사업 타겟 — 내부 앱 vs 상세 페이지 vs 삼성

#### 논의 내용

- **Attendees 3**: 타겟을 하나 잡고 목표를 정해야 싱크를 맞출 수 있음. 마냥 "서치해봐, 준비해봐"는 곤란.
- **Attendees 1**: 타겟은 내부 앱을 정하면 됨. 지금 저쪽에서 하고 있는 것도 몇 개 있음.
- **Attendees 2**: 과거부터 상세 페이지 니즈가 지속적으로 있었음. 소상공인은 돈이 안 되지만, 플랫폼사(롯데ON, G마켓 등)는 SKU가 많아 대량 생성에 관심 높음. 템플릿 1개로 찍어내는 게 유의미.
- **Attendees 5 (다봄)**: 처음 만들 때부터 상세 페이지 생각하고 만들었음. 일단 사내에서 쓸 수 있게끔.
- **Attendees 2**: 고객은 개인보다 플랫폼일 가능성 높음. 신규 사업팀에서 요구사항 구체적으로 정리 필요.
- **Attendees 1**: 이미지/미디어 서빙이 고민. 다 만들고 누가 쓴다고 할 때 고민하면 됨.
- **Attendees 2**: 고객사 CDN API로 보내고 발급받아 교체하는 방법도 있음.

#### 결정 사항

- 1차 타겟: 내부 앱 / 사내 상세 페이지
- 2차 타겟: 플랫폼사 (G마켓, 롯데ON 등) 대량 상세 페이지 생성
- 3차 타겟: 삼성닷컴 로컬라이제이션
- 이미지/미디어 서빙 문제는 후순위 (사용자가 생기면 그때 고민)

### 안건 5: 기술 구조 — 섹션 트리, CMS 연결, Export

#### 논의 내용

- **Attendees 1**: 현재 DOM 트리를 섹션 트리로 단순화해야 함. 고객사별로 섹션을 명확하게 정의하고 조합. 나모에디터처럼 개발자/디자이너용이 아니라 비개발자도 쓸 수 있게 단순화.
- **Attendees 2**: 성공하려면 시맨틱한 부분(플랫폼 CMS 연결)이 핵심. 고객 입장에서는 정보만 관심 있지, 디자인은 한번 코드로 넘어오면 끝.
- **Attendees 3**: "세일 프라이스" 텍스트 영역 선택하면 CMS 필드와 연결되는 개념?
- **Attendees 1**: 시나리오 많음 — 외부 업체 API로 상품 정보 연결, 훅으로 바로 페이지 생성 등.
- **Attendees 1**: Export는 메이저 업체별로 플러그인처럼 붙이면 됨. 차트, 인포그래픽, AI 생성 이미지/영상도 섹션으로 넣을 수 있음.

#### 결정 사항

- DOM 트리 → 섹션 트리로 추상화 필요
- 고객사별 CMS 연결(데이터 바인딩)이 핵심 차별점
- Export는 플랫폼별 플러그인 구조로 설계
- AI 기반 콘텐츠 생성(이미지, 영상, 인포그래픽)은 스텝 1-2 이후 확장

### 안건 6: 로직 삽입 기술 방향

#### 논의 내용

- **Attendees 1**: 로직을 넣으려면 다이나믹 번들링 필요. Sandpack 또는 유사 도구. 워크플로우가 생기면 복잡해짐 — 화면 구성 파트와 워크플로우 파트를 분리해야 함. 워크플로우는 백엔드에서 처리하는 게 안전.
- **Attendees 5 (다봄)**: 나중에 로직을 넣는 부분은 어떻게 해야 할지 고민.
- **Attendees 1**: 에이전트도 붙어야 함. 서비스가 계속 늘어날 것이므로. 재환님의 AX 리포 참고.

#### 결정 사항

- Sandpack 등 다이나믹 번들링으로 로직 삽입 접근
- 허용된 API/기능만 사용 가능하도록 강하게 제한
- 워크플로우는 백엔드 주도로 처리
- AI 에이전트 연동 필요 (재환님 AX 리포 참고)
